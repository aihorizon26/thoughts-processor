# Vercel Deployment Guide for Thoughts App

This guide explains how to deploy the Thoughts App to Vercel with a real Supabase backend.

## Prerequisites

1. **Supabase Account**: You need a Supabase project with:
   - Project URL (found in Settings → API)
   - Anon/public key (found in Settings → API)

2. **Vercel Account**: You need a Vercel account connected to your GitHub repository.

3. **GitHub Repository**: Your code should be in a GitHub repository that Vercel can access.

## Step-by-Step Deployment Instructions

### Step 1: Prepare Your Supabase Database

1. Log into your Supabase dashboard at https://supabase.com/dashboard
2. Select your project or create a new one
3. Go to the SQL editor and run this SQL to create the thoughts table:

```sql
create table thoughts (
  id bigint generated by default as identity primary key,
  thought text not null,
  classification text not null,
  roi double precision not null,
  created_at timestamp with time zone default timezone('utc' now()) not null
);

-- Enable real-time (optional, but useful for future enhancements)
alter publication supabase_realtime add table thoughts;
```

4. (Optional) Set up Row Level Security if you want to restrict access:
   - For a public API where anyone can read/write, you can disable RLS:
     ```sql
     alter table thoughts disable row level security;
     ```
   - For more secure access, you'll need to implement authentication (beyond the scope of this basic guide).

### Step 2: Configure Environment Variables in Vercel

1. In Vercel dashboard, go to your project → Settings → Environment Variables
2. Add two environment variables:
   - **Name**: `SUPABASE_URL`
     - **Value**: Your Supabase project URL (e.g., `https://xyzcompany.supabase.co`)
     - **Environment**: Production (and Preview if you want deployments from preview branches)
   - **Name**: `SUPABASE_KEY`
     - **Value**: Your Supabase anon/public key
     - **Environment**: Production (and Preview if needed)

> **Important**: Do NOT use the service_role key here - use the anon/public key only. The service_role key bypasses all security and should only be used in trusted server environments.

### Step 3: Verify Vercel Configuration

The `vercel.json` file should already be correctly configured:

```json
{
  "version": 2,
  "builds": [{ "src": "api/index.py", "use": "@vercel/python" }],
  "routes": [{ "src": "/(.*)", "dest": "api/index.py" }]
}
```

This tells Vercel:
- Use the Python builder for `api/index.py`
- Route all requests to that API file

### Step 4: Deploy to Vercel

1. Push your code to GitHub (if you haven't already)
2. In Vercel dashboard, click "New Project"
3. Import your GitHub repository
4. Vercel should automatically detect it's a Python project
5. Click "Deploy"
6. Vercel will:
   - Install dependencies from `requirements.txt`
   - Build your application using `@vercel/python`
   - Deploy the API endpoints

### Step 5: Test Your Deployment

Once deployment succeeds, you'll get a URL like `https://your-project-name.vercel.app`

Test the endpoints:

**Get all thoughts:**
```bash
curl https://your-project-name.vercel.app/api/thoughts
```

**Add a new thought:**
```bash
curl -X POST https://your-project-name.vercel.app/api/thoughts \
  -H "Content-Type: application/json" \
  -d '{"thought": "Test thought from Vercel deployment"}'
```

### Step 6: Troubleshooting Common Issues

#### Issue 1: ModuleNotFoundError for supabase
- **Solution**: Make sure `supabase` is in your `requirements.txt`. For this app, we removed it to avoid build issues and use a mock when unavailable. For production with real Supabase, you need to add it back:
  ```
  flask
  python-dotenv
  supabase
  ```

#### Issue 2: Connection errors to Supabase
- **Solution**: Double-check:
  - Your SUPABASE_URL and SUPABASE_KEY are correct in Vercel environment variables
  - Your Supabase project is active (not paused)
  - You're using the anon/public key, not the service_role key
  - Network connectivity (Supabase should be accessible from Vercel's AWS regions)

#### Issue 3: Table doesn't exist errors
- **Solution**: Verify:
  - You ran the SQL to create the `thoughts` table in your Supabase database
  - The table name is exactly `thoughts` (case-sensitive in some contexts)
  - You have permissions to read/write to the table

#### Issue 4: Build fails due to pyiceberg or other dependencies
- **Solution**: The current `requirements.txt` is deliberately minimal to avoid build issues:
  ```
  flask
  python-dotenv
  ```
  If you need the real Supabase client for production, update it to:
  ```
  flask
  python-dotenv
  supabase
  ```
  And make sure you have a proper build environment (Vercel's Python builder should handle this).

### Local Development with Real Supabase

To test locally with your real Supabase credentials (before deploying):

```bash
# Set environment variables
set SUPABASE_URL=your_supabase_url_here
set SUPABASE_KEY=your_supabase_anon_key_here

# Run the API
python api/index.py

# Test in another terminal:
curl http://localhost:5000/api/thoughts
curl -X POST http://localhost:5000/api/thoughts -H "Content-Type: application/json" -d '{"thought": "local test"}'
```

### How the App Handles Missing Supabase

The app intelligently falls back to a mock Supabase client when:
- Environment variables are missing
- The supabase module cannot be imported
- Any error occurs during client initialization

This means:
- Local development works without Supabase (uses `mock_thoughts.json`)
- Production deployment works with real Supabase when variables are set
- You get graceful degradation rather than hard crashes
